# Conduit — Operation Bus Framework  
### Техническая спецификация v1.0  

> **Почему Conduit:** conduit — это канал, по которому проходит что-то важное, не трансформируясь. Именно это делает фреймворк: операция входит с одной стороны, выходит с другой — независимо от того, что находится между ними.  

---  

## 1. Назначение и область применения  

Conduit — это **архитектурный фреймворк** унификации межкомпонентного взаимодействия в backend-системах.  

Он определяет **единый контракт и предсказуемую модель поведения** для двух классов операций — команд и событий — независимо от того, где физически исполняется получатель: в том же процессе, в соседнем сервисе или за брокером сообщений.  

> **Чем Conduit не является:**  
> - Не сетевой протокол с wire format и handshake  
> - Не замена Kafka, RabbitMQ или gRPC  
> - Не RPC-фреймворк для запросов данных  
> - Не новый механизм — комбинация проверенных паттернов с единым контрактом  
>  
> Conduit — это **слой архитектурных соглашений и маршрутизации**, поверх которого работают существующие транспорты.  

**Область применения:** любая backend-система, в которой два или более компонента обмениваются данными или координируют действия — вне зависимости от текущей топологии развёртывания.  

> **Ключевой принцип:** код, инициирующий операцию, не знает и не должен знать о топологии исполнения. Это знание вынесено на уровень конфигурации маршрута.  

---  

## 2. Проблематика  

### 2.1 Архитектурная дилемма роста  

Большинство систем начинаются как монолит и в какой-то момент сталкиваются с необходимостью декомпозиции. Переход на микросервисы требует переработки коммуникаций: прямые вызовы заменяются HTTP/gRPC, синхронные операции — очередями, появляются новые слои надёжности.  

Это приводит к **переписыванию бизнес-логики** не по причине её изменения, а исключительно по причине смены транспорта — структурные изменения без продуктовой ценности.  

### 2.2 Фрагментация инструментов  

Существующие инструменты решают задачу частично:  

| Инструмент | Что решает | Что не решает |  
|---|---|---|  
| Kafka / RabbitMQ / NATS | Надёжный транспорт сообщений | Унификацию семантики вызовов |  
| CQRS-фреймворки | Разделение Command и Event | Транспортную абстракцию |  
| gRPC / REST | Стандарт сетевого вызова | Асинхронность, Outbox, локальный режим |  
| MediatR / EventBus | Локальный диспатч | Сетевой транспорт, гарантии доставки |  
| Debezium / Outbox libs | Надёжная доставка событий | Унификацию типов операций |  

Conduit **не изобретает** новые механизмы. Он берёт известные паттерны — CQRS, Transactional Outbox, Schema Registry, OpenTelemetry — и объединяет их в единую согласованную модель с предсказуемым поведением.  

### 2.3 Отсутствие встроенной надёжности  

В большинстве систем гарантии доставки добавляются постфактум. Это приводит к race condition между изменением данных и публикацией события — классическая проблема, которая решается вручную в каждом проекте по-разному.

## 3. Концептуальная модель  

### 3.1 Место в стеке  

```  
┌──────────────────────────────────────────────┐  
│           Бизнес-логика                      │  
├──────────────────────────────────────────────┤  
│   Conduit Bus                                │  ← соглашения, контракты,  
│   (маршрутизация операций)                   │    маршрутизация, политики  
├──────────────────────────────────────────────┤  
│  InMemory │ Outbox/DB │ Kafka │ NATS │ ...   │  ← физический транспорт  
├──────────────────────────────────────────────┤  
│           Сеть / процесс / БД                │  
└──────────────────────────────────────────────┘  
```  

Conduit не заменяет Kafka или gRPC — он абстрагирует выбор между ними для операционного слоя.  

### 3.2 Уровни абстракции  

```  
┌────────────────────────────────────────────────────────┐  
│  L3 — Бизнес-логика                                    │  
│       dispatch(operation) — единственный вызов         │  
├────────────────────────────────────────────────────────┤  
│  L2 — Conduit Router                                   │  
│       Маршрут, провайдер, политики, версионирование    │  
├───────────────┬────────────────┬───────────────────────┤  
│  L1a          │  L1b           │  L1c                  │  
│  InMemory     │  Outbox/DB     │  Broker               │  
│  Provider     │  Provider      │  Provider             │  
└───────────────┴────────────────┴───────────────────────┘  
```  

### 3.3 Область ответственности: что входит, что нет  

```  
┌───────────────────────────────────┬───────────────────────────────────┐  
│  Входит в Conduit                 │  Не входит в Conduit              │  
├───────────────────────────────────┼───────────────────────────────────┤  
│  Command — изменение состояния    │  Query / RPC — запросы данных     │  
│  Event — уведомление о факте      │  Streaming — потоки байт          │  
│  Маршрутизация и провайдеры       │  CRUD API для внешних клиентов    │  
│  Outbox и политики повторов       │  Real-time UI push                │  
│  Версионирование операций         │  Batch-обработка                  │  
│  Трассировка и метрики            │                                   │  
└───────────────────────────────────┴───────────────────────────────────┘  
```  

> **Почему Query исключён из Conduit:**  
> Query — это синхронный RPC с типизированным ответом. Прогонять его через инфраструктуру асинхронных операций означает добавлять latency и усложнение без выгоды. Кроме того, это противоречит духу CQRS: read path и write path должны быть физически разделены, а не объединены в одном роутере.  
>  
> Для внутрипроцессных запросов используйте прямые вызовы (in-process DI). Для межсервисных — gRPC или REST. Conduit обслуживает только операции с side effects и уведомления.

## 4. Типология операций  

Conduit работает с двумя семантически различными и непересекающимися классами операций.  

### 4.1 Command — команда  

**Семантика:** императив. Инициирует изменение состояния системы.  

```  
┌────────────────────────────────────────────────────────────┐  
│  COMMAND                                                   │  
├─────────────────────┬──────────────────────────────────────┤  
│  Направленность     │  Один отправитель → один получатель  │  
├─────────────────────┼──────────────────────────────────────┤  
│  Ожидание ответа    │  Подтверждение (success / failure)   │  
│                     │  Синхронно или через correlation      │  
├─────────────────────┼──────────────────────────────────────┤  
│  Семантика доставки │  At-least-once                       │  
├─────────────────────┼──────────────────────────────────────┤  
│  Эффект             │  Единственный — при обязательной     │  
│                     │  идемпотентности обработчика         │  
├─────────────────────┼──────────────────────────────────────┤  
│  Изменяет состояние │  Да                                  │  
├─────────────────────┼──────────────────────────────────────┤  
│  Примеры            │  CreateOrder, CancelPayment,         │  
│                     │  AssignUser                          │  
└─────────────────────┴──────────────────────────────────────┘  
```  

**Гарантия доставки — явное определение:**  

Conduit **не обеспечивает exactly-once на уровне инфраструктуры**. Это физически невозможно без распределённых транзакций (2PC). Реальная модель:  

```  
Conduit гарантирует:  
  at-least-once доставку (через Outbox или брокер)  

Обработчик обязан гарантировать:  
  идемпотентность через дедупликацию по idempotency_key  

Результат:  
  единственный видимый эффект для бизнес-логики —  
  но только при корректной реализации обработчика  
```  

Это **контрактное требование к обработчику**, а не гарантия фреймворка. Если обработчик не реализует дедупликацию — повторная доставка приведёт к двойному исполнению.  

```  
Пример дедупликации на стороне обработчика:  

  BEGIN TRANSACTION  
    IF EXISTS (SELECT 1 FROM processed_ops  
               WHERE idempotency_key = :key)  
      RETURN AlreadyProcessed  
    END  
    -- выполнить бизнес-логику  
    INSERT INTO processed_ops (idempotency_key, processed_at)  
  COMMIT  
```  

### 4.2 Event — событие  

**Семантика:** факт. Уведомляет о произошедшем изменении.  

```  
┌────────────────────────────────────────────────────────────┐  
│  EVENT                                                     │  
├─────────────────────┬──────────────────────────────────────┤  
│  Направленность     │  Один отправитель → N подписчиков   │  
│                     │  (0..∞)                              │  
├─────────────────────┼──────────────────────────────────────┤  
│  Ожидание ответа    │  Нет                                 │  
├─────────────────────┼──────────────────────────────────────┤  
│  Семантика доставки │  At-least-once                       │  
├─────────────────────┼──────────────────────────────────────┤  
│  Изменяет состояние │  Нет (у отправителя)                 │  
├─────────────────────┼──────────────────────────────────────┤  
│  Примеры            │  OrderCreated, PaymentFailed,        │  
│                     │  UserRegistered                      │  
└─────────────────────┴──────────────────────────────────────┘  
```  

**Инвариант:** отправитель не знает о подписчиках и не зависит от их наличия. Событие с нулём подписчиков — валидное состояние.

## 5. Контракт операции  

### 5.1 Envelope — конверт операции  

```  
┌──────────────────────┬──────────────────────────────────────┐  
│  operation_id        │  UUID v7 — уникальный идентификатор  │  
│                      │  операции. v7 выбран намеренно:      │  
│                      │  монотонно возрастает, пригоден      │  
│                      │  для сортировки по времени           │  
├──────────────────────┼──────────────────────────────────────┤  
│  operation_type      │  COMMAND | EVENT                     │  
├──────────────────────┼──────────────────────────────────────┤  
│  operation_name      │  Строка в формате domain.action      │  
│                      │  Например: "order.create",           │  
│                      │  "payment.failed"                    │  
├──────────────────────┼──────────────────────────────────────┤  
│  schema_version      │  Semver строка: "1.2.0"              │  
│                      │  Версия схемы payload                │  
├──────────────────────┼──────────────────────────────────────┤  
│  payload             │  Тело операции.                      │  
│                      │  JSON / Protobuf / MessagePack       │  
├──────────────────────┼──────────────────────────────────────┤  
│  metadata            │  См. раздел 5.2                      │  
├──────────────────────┼──────────────────────────────────────┤  
│  created_at          │  ISO 8601 timestamp (UTC)            │  
├──────────────────────┼──────────────────────────────────────┤  
│  expires_at          │  Опционально. TTL операции.          │  
│                      │  По истечении — в DLQ без попыток   │  
└──────────────────────┴──────────────────────────────────────┘  
```  

### 5.2 Metadata — системные метаданные  

```  
┌──────────────────────┬──────────────────────────────────────┐  
│  trace_id            │  Сквозной идентификатор              │  
│                      │  трассировки (OpenTelemetry W3C      │  
│                      │  Trace Context)                      │  
├──────────────────────┼──────────────────────────────────────┤  
│  correlation_id      │  Связь с исходной операцией.         │  
│                      │  Для асинхронных Command-ответов     │  
│                      │  и цепочек событий                   │  
├──────────────────────┼──────────────────────────────────────┤  
│  causation_id        │  operation_id операции-причины.      │  
│                      │  Позволяет восстановить              │  
│                      │  причинно-следственную цепочку       │  
├──────────────────────┼──────────────────────────────────────┤  
│  source_service      │  Идентификатор сервиса-отправителя.  │  
│                      │  Используется для авторизации        │  
│                      │  и observability                     │  
├──────────────────────┼──────────────────────────────────────┤  
│  idempotency_key     │  Ключ дедупликации на стороне        │  
│                      │  обработчика. Обязателен для         │  
│                      │  COMMAND. Для EVENT — опционален     │  
├──────────────────────┼──────────────────────────────────────┤  
│  priority            │  LOW | NORMAL | HIGH | CRITICAL.     │  
│                      │  Интерпретируется провайдером,       │  
│                      │  если поддерживается                 │  
├──────────────────────┼──────────────────────────────────────┤  
│  attempt_number      │  Номер попытки доставки (1..N).      │  
│                      │  Проставляется фреймворком,          │  
│                      │  не отправителем                     │  
├──────────────────────┼──────────────────────────────────────┤  
│  reply_to            │  Опционально. Для асинхронного       │  
│                      │  Command: имя операции или топик,    │  
│                      │  куда доставить ответ                │  
├──────────────────────┼──────────────────────────────────────┤  
│  headers             │  Произвольные key-value пары         │  
│                      │  для расширений и middleware         │  
└──────────────────────┴──────────────────────────────────────┘  
```

## 6. Асинхронный ответ на Command  

При синхронной доставке (InMemory) ответ возвращается немедленно. При асинхронной — модель ответа требует явного определения.  

### 6.1 Модели получения ответа  

```  
┌─────────────────────────────────────────────────────────────┐  
│  Модель 1: Fire-and-confirm (синхронная)                    │  
│                                                             │  
│  dispatch(cmd) ──► Handler ──► Result                       │  
│                                                             │  
│  Применение: InMemory провайдер                             │  
│  Гарантия: результат известен сразу                         │  
└─────────────────────────────────────────────────────────────┘  

┌─────────────────────────────────────────────────────────────┐  
│  Модель 2: Correlation-based (асинхронная)                  │  
│                                                             │  
│  1. Отправитель публикует Command с:                        │  
│     - operation_id: "op-123"                               │  
│     - reply_to: "order.create.reply"                       │  
│                                                             │  
│  2. Handler исполняет команду, публикует Event:             │  
│     - operation_name: "order.create.reply"                 │  
│     - correlation_id: "op-123"                             │  
│     - payload: { status: "ok", order_id: "..." }           │  
│                                                             │  
│  3. Отправитель подписан на "order.create.reply",          │  
│     матчит по correlation_id                               │  
│                                                             │  
│  Применение: Outbox / Kafka провайдер                       │  
│  Гарантия: ответ доставлен at-least-once                    │  
└─────────────────────────────────────────────────────────────┘  

┌─────────────────────────────────────────────────────────────┐  
│  Модель 3: Fire-and-forget (без ожидания ответа)            │  
│                                                             │  
│  dispatch(cmd) ──► Outbox ──► (исполнится когда-нибудь)    │  
│                                                             │  
│  Применение: некритичные команды без немедленного фидбека   │  
│  Гарантия: at-least-once доставка                           │  
└─────────────────────────────────────────────────────────────┘  
```  

Модель выбирается в конфигурации маршрута. Если `reply_to` не указан — маршрут работает в режиме fire-and-forget.

## 7. Транспортные провайдеры  

### 7.1 Матрица провайдеров  

```  
┌──────────────┬──────────────┬────────────────┬──────────────┬────────────────┐  
│  Провайдер   │  Топология   │  Гарантия      │  Латентность │  Применение    │  
├──────────────┼──────────────┼────────────────┼──────────────┼────────────────┤  
│  InMemory    │  In-process  │  Best-effort   │  ~0          │  Тесты,        │  
│              │              │                │              │  dev-среда,    │  
│              │              │                │              │  монолит       │  
├──────────────┼──────────────┼────────────────┼──────────────┼────────────────┤  
│  Outbox/DB   │  In-process  │  At-least-once │  Низкая (ms) │  Продакшн,     │  
│              │  + async     │                │              │  надёжные      │  
│              │  relay       │                │              │  команды и     │  
│              │              │                │              │  события       │  
├──────────────┼──────────────┼────────────────┼──────────────┼────────────────┤  
│  Kafka       │  Distributed │  At-least-once │  Низкая (ms) │  Высокая       │  
│              │              │                │              │  нагрузка,     │  
│              │              │                │              │  стриминг      │  
├──────────────┼──────────────┼────────────────┼──────────────┼────────────────┤  
│  RabbitMQ    │  Distributed │  At-least-once │  Низкая (ms) │  Task queues,  │  
│              │              │                │              │  routing       │  
├──────────────┼──────────────┼────────────────┼──────────────┼────────────────┤  
│  NATS Core   │  Distributed │  Best-effort   │  Очень       │  Real-time,    │  
│              │              │                │  низкая (μs) │  IoT, edge     │  
├──────────────┼──────────────┼────────────────┼──────────────┼────────────────┤  
│  NATS        │  Distributed │  At-least-once │  Низкая (ms) │  Real-time     │  
│  JetStream   │              │                │              │  + гарантии    │  
└──────────────┴──────────────┴────────────────┴──────────────┴────────────────┘  
```  

### 7.2 Что меняется при смене провайдера  

Смена провайдера — это изменение конфигурации, не кода бизнес-логики. Однако разработчик **обязан понимать**, что при этом меняется:  

```  
┌──────────────────────┬──────────────────┬──────────────────────┐  
│  Аспект              │  InMemory        │  Kafka               │  
├──────────────────────┼──────────────────┼──────────────────────┤  
│  Ошибки              │  исключение      │  таймаут /           │  
│                      │  немедленно      │  недоступность        │  
│                      │                  │  брокера             │  
├──────────────────────┼──────────────────┼──────────────────────┤  
│  Транзакционность    │  в рамках        │  Outbox + отдельная  │  
│                      │  текущей         │  транзакция relay    │  
├──────────────────────┼──────────────────┼──────────────────────┤  
│  Порядок доставки    │  гарантирован    │  гарантирован        │  
│                      │                  │  внутри partition    │  
├──────────────────────┼──────────────────┼──────────────────────┤  
│  Откат               │  вместе с        │  независимо от       │  
│  транзакции          │  транзакцией     │  бизнес-транзакции   │  
└──────────────────────┴──────────────────┴──────────────────────┘  
```  

Абстракция скрывает детали транспорта, но **не скрывает его природу**. Тесты на InMemory не заменяют интеграционные тесты с реальным брокером.  

### 7.3 Конфигурация маршрутов  

```  
route("order.create")  
  .type(COMMAND)  
  .via(Provider.OUTBOX)  
  .withRetry(attempts: 3, backoff: EXPONENTIAL)  
  .withTimeout(seconds: 30)  
  .replyTo("order.create.reply")          // correlation-based ответ  
  .onExhausted(DLQ)  

route("payment.failed")  
  .type(EVENT)  
  .via(Provider.KAFKA)  
  .withPriority(CRITICAL)  
  .withRetry(attempts: 5, backoff: EXPONENTIAL)  
  .onExhausted(DLQ)  

route("inventory.updated")  
  .type(EVENT)  
  .via(Provider.IN_MEMORY)  
  .withTimeout(seconds: 1)  
  .onExhausted(LOG_AND_DROP)             // некритичное событие  
```

## 8. Маршрутизация операций  

### 8.1 Жизненный цикл операции  

```  
  Инициатор  
      │  
      │  dispatch(Envelope)  
      ▼  
┌─────────────────────────┐  
│      Conduit Router     │  
│                         │  
│  1. Валидация envelope  │  
│  2. Проверка TTL        │  
│  3. Lookup маршрута     │  
│  4. Авторизация         │  
│     source_service      │  
│  5. Выбор провайдера    │  
│  6. Применение политик  │  
│  7. Backpressure check  │  
└──────────┬──────────────┘  
           │  
           ├────────────────────────────────┐  
           │                                │  
           ▼                                ▼  
   [InMemory Provider]            [Async Provider]  
           │                                │  
           │  прямой вызов                  │  сериализация  
           ▼                                │  + запись / публикация  
    ┌─────────────┐                         ▼  
    │   Handler   │                ┌─────────────────┐  
    └──────┬──────┘                │  Outbox / Broker│  
           │                       └────────┬────────┘  
           │  Result / Error                │  
           │                                │  фоновая доставка  
           ▼                                ▼  
      Инициатор                    ┌─────────────┐  
      (немедленно)                 │   Handler   │  
                                   └──────┬──────┘  
                                          │  
                                          │  correlation-based  
                                          │  ответ (если reply_to)  
                                          ▼  
                                     Инициатор  
```  

### 8.2 Backpressure  

При перегрузке Conduit Router применяет явную политику backpressure вместо молчаливого накопления:  

```  
┌──────────────────────────────────────────────────────────────┐  
│  Условие                  │  Поведение по умолчанию          │  
├───────────────────────────┼──────────────────────────────────┤  
│  Outbox: > N PENDING       │  dispatch() возвращает          │  
│  записей (watermark)      │  BackpressureException           │  
├───────────────────────────┼──────────────────────────────────┤  
│  Handler: достигнут       │  Router блокирует dispatch()     │  
│  concurrency limit        │  на configurable timeout,        │  
│                           │  затем — исключение              │  
├───────────────────────────┼──────────────────────────────────┤  
│  Broker недоступен        │  InMemory-буфер на N операций,   │  
│                           │  при переполнении — исключение   │  
└───────────────────────────┴──────────────────────────────────┘  
```  

Конфигурация backpressure на маршруте:  
```  
route("order.create")  
  .withBackpressure(  
    outboxWatermark: 10_000,     // макс. PENDING записей  
    blockTimeout: seconds(5),    // ждать освобождения  
    onOverflow: RAISE_EXCEPTION  // или: DROP, SAMPLE  
  )  
```  

### 8.3 Модель подписки  

```  
┌──────────────────────┬─────────────────────────────────────┐  
│  operation_name      │  Имя операции или паттерн           │  
│                      │  "order.*" — все события домена     │  
├──────────────────────┼─────────────────────────────────────┤  
│  operation_type      │  COMMAND | EVENT                    │  
├──────────────────────┼─────────────────────────────────────┤  
│  schema_version      │  Semver range: ">=1.0.0 <2.0.0"    │  
├──────────────────────┼─────────────────────────────────────┤  
│  consumer_group      │  Для EVENT: группа потребителей     │  
├──────────────────────┼─────────────────────────────────────┤  
│  concurrency         │  Макс. параллельных вызовов         │  
├──────────────────────┼─────────────────────────────────────┤  
│  handler_ref         │  Ссылка на обработчик               │  
└──────────────────────┴─────────────────────────────────────┘  
```  

**Модели доставки событий:**  

```  
  Fan-out (broadcast):              Competing consumers:  

  Event ──► Handler A               Event ──► [Group]──► Handler A  
        ├──► Handler B                                └──► Handler B  
        └──► Handler C              (только один получит)  
  (все получают копию)  
```

## 9. Модель надёжности  

### 9.1 Transactional Outbox  

Для операций, требующих гарантированной доставки:  

```  
┌──────────────────────────────────────────────────────────────┐  
│  Бизнес-транзакция (атомарно)                                │  
│  ┌──────────────────────┐   ┌──────────────────────────┐    │  
│  │  Изменение доменных  │ + │  INSERT INTO outbox       │    │  
│  │  данных              │   │  (envelope,               │    │  
│  └──────────────────────┘   │   status = PENDING,       │    │  
│                              │   created_at,             │    │  
│                              │   partition_key)          │    │  
│                              └──────────────────────────┘    │  
└──────────────────────────────────────────────────────────────┘  
                  │  
                  │  (транзакция завершена)  
                  ▼  
┌──────────────────────────────────────────────────────────────┐  
│  Outbox Relay (фоновый процесс)                              │  
│                                                              │  
│  1. SELECT ... FROM outbox                                   │  
│     WHERE status = PENDING                                   │  
│     ORDER BY created_at                                      │  
│     LIMIT :batch_size                                        │  
│     FOR UPDATE SKIP LOCKED      ← конкурентная безопасность │  
│                                                              │  
│  2. Publish → Transport Provider                             │  
│     (с подтверждением от брокера)                           │  
│                                                              │  
│  3. UPDATE outbox                                            │  
│     SET status = DELIVERED,                                  │  
│         delivered_at = now()                                 │  
│     WHERE operation_id IN (...)                              │  
│                                                              │  
│  При ошибке публикации:                                      │  
│     UPDATE outbox                                            │  
│     SET attempt_number = attempt_number + 1,                 │  
│         next_attempt_at = now() + backoff(attempt_number),   │  
│         last_error = :error                                  │  
└──────────────────────────────────────────────────────────────┘  
```  

**Relay: ключевые параметры конфигурации:**  

```  
┌─────────────────────┬────────────────────────────────────────┐  
│  poll_interval      │  Интервал опроса таблицы               │  
│                     │  (например: 100ms)                     │  
├─────────────────────┼────────────────────────────────────────┤  
│  batch_size         │  Число записей за одну итерацию        │  
│                     │  (например: 100)                       │  
├─────────────────────┼────────────────────────────────────────┤  
│  relay_instances    │  Число параллельных Relay.             │  
│                     │  FOR UPDATE SKIP LOCKED обеспечивает   │  
│                     │  безопасность при N > 1                │  
├─────────────────────┼────────────────────────────────────────┤  
│  ordering_key       │  partition_key в записи Outbox.        │  
│                     │  Порядок гарантирован внутри           │  
│                     │  одного ключа (например: order_id)     │  
├─────────────────────┼────────────────────────────────────────┤  
│  retention          │  Срок хранения DELIVERED записей       │  
│                     │  до очистки (например: 7 дней)         │  
└─────────────────────┴────────────────────────────────────────┘  
```  

**Гарантия порядка:**  

```  
Outbox гарантирует порядок только внутри одного partition_key.  

  partition_key = "order-123":  
    [1] OrderCreated    ──► доставлен первым   ✅  
    [2] OrderConfirmed  ──► доставлен вторым   ✅  
    [3] OrderShipped    ──► доставлен третьим  ✅  

  Разные partition_key — порядок между ними не гарантирован.  
```  

### 9.2 Уровни гарантий доставки  

```  
┌─────────────────┬────────────────────────────────────────────┐  
│  Best-effort    │  Без подтверждений. Потеря при сбое        │  
│                 │  возможна. Применение: dev, некритичные    │  
│                 │  события, NATS Core                        │  
├─────────────────┼────────────────────────────────────────────┤  
│  At-least-once  │  Гарантия доставки минимум один раз.       │  
│                 │  Повторы возможны. Требует идемпотентного  │  
│                 │  обработчика. Применение: большинство      │  
│                 │  бизнес-операций                           │  
├─────────────────┼────────────────────────────────────────────┤  
│  Effectively    │  Не гарантия фреймворка, а архитектурный   │  
│  once           │  результат: at-least-once доставка +       │  
│                 │  корректная дедупликация в обработчике.    │  
│                 │  Применение: платежи, финансовые операции  │  
└─────────────────┴────────────────────────────────────────────┘  
```  

### 9.3 Политики повторов  

```  
┌─────────────────────┬──────────────────────────────────────────┐  
│  max_attempts       │  Максимум попыток (1..N или ∞)          │  
├─────────────────────┼──────────────────────────────────────────┤  
│  backoff_strategy   │  FIXED | LINEAR | EXPONENTIAL            │  
├─────────────────────┼──────────────────────────────────────────┤  
│  initial_delay      │  Задержка перед первым повтором          │  
├─────────────────────┼──────────────────────────────────────────┤  
│  max_delay          │  Потолок задержки (для EXPONENTIAL)      │  
├─────────────────────┼──────────────────────────────────────────┤  
│  jitter             │  Рандомизация задержки. Обязательна      │  
│                     │  при N > 1 параллельных consumer:        │  
│                     │  предотвращает thundering herd           │  
├─────────────────────┼──────────────────────────────────────────┤  
│  retry_on           │  Классы ошибок для повтора.              │  
│                     │  По умолчанию: все recoverable errors.   │  
│                     │  Исключение: бизнес-ошибки (например     │  
│                     │  ValidationError) — не повторяются       │  
├─────────────────────┼──────────────────────────────────────────┤  
│  on_exhausted       │  DLQ | LOG_AND_DROP | RAISE              │  
└─────────────────────┴──────────────────────────────────────────┘  
```  

### 9.4 Dead Letter Queue  

```  
  Обычный поток:  
  Outbox ──► Provider ──► Handler ──► ✅ Доставлено  

  Поток с ошибками:  
  Outbox ──► Provider ──► Handler ──► ❌ Error  
                                    ──► retry 2 (backoff)  
                                    ──► ❌ Error  
                                    ──► retry 3 (backoff)  
                                    ──► ❌ Error  
                                    ──► DLQ  

  DLQ содержит:  
  - Оригинальный Envelope (без изменений)  
  - Число попыток  
  - Последняя ошибка + stacktrace  
  - Временные метки всех попыток  

  DLQ — это не корзина. Это очередь для ручного разбора,  
  алертинга и возможного replay после устранения причины.  
```

## 10. Версионирование операций  

### 10.1 Стратегии совместимости  

```  
┌─────────────────────────────────────────────────────────────┐  
│  Backward Compatible (предпочтительно)                      │  
│                                                             │  
│  Правила:                                                   │  
│  - Новые поля только опциональные                           │  
│  - Старый обработчик игнорирует неизвестные поля            │  
│  - Версия: 1.0.0 → 1.1.0                                   │  
├─────────────────────────────────────────────────────────────┤  
│  Breaking Change                                            │  
│                                                             │  
│  Правила:                                                   │  
│  - Удаление / переименование полей                          │  
│  - Смена типов                                              │  
│  - Версия: 1.x.x → 2.0.0                                   │  
│  - Оба обработчика работают параллельно                     │  
└─────────────────────────────────────────────────────────────┘  
```  

### 10.2 Алгоритм маршрутизации по версии  

```  
  Envelope приходит с schema_version: "1.3.0"  

  Conduit Router выполняет:  

  1. Собрать всех кандидатов для operation_name  

  2. Отфильтровать по semver range:  
     Handler_A: ">=1.0.0 <2.0.0"  →  1.3.0 входит  ✅  
     Handler_B: ">=2.0.0 <3.0.0"  →  1.3.0 не входит ❌  

  3. Если кандидатов > 1 (для COMMAND — ошибка конфигурации):  
     COMMAND: один обработчик обязателен →  
              ConfigurationException при старте  
     EVENT:   все подходящие получают копию (fan-out)  
              или один из группы (competing consumers)  

  4. Если кандидатов = 0:  
     expires_at ещё не прошёл → в DLQ с NoHandlerError  
     expires_at прошёл        → LOG_AND_DROP  

  5. Версия из envelope не изменяется при доставке.  
     Handler видит оригинальную версию.  
```  

### 10.3 Параллельное версионирование  

```  
  Переход с v1 на v2:  

  ┌──────────────┐   operation (v1.x)   ┌──────────────┐  
  │  Producer A  │─────────────────────►│  Handler v1  │  
  │  (старый)    │                       │  >=1.0 <2.0  │  
  └──────────────┘                       └──────────────┘  

  ┌──────────────┐   operation (v2.x)   ┌──────────────┐  
  │  Producer B  │─────────────────────►│  Handler v2  │  
  │  (новый)     │                       │  >=2.0 <3.0  │  
  └──────────────┘                       └──────────────┘  

  Оба работают одновременно без конфликтов.  
  Переход завершается, когда Producer A мигрирован.  
  Handler v1 снимается только после этого.  
```

## 11. Observability  

### 11.1 Распределённая трассировка  

Conduit интегрируется с **OpenTelemetry** по стандарту W3C Trace Context. Каждая операция автоматически создаёт span:  

```  
  trace_id: abc-123  
  │  
  ├── [span] conduit.dispatch: order.create (api-gateway)  
  │       operation_id: op-456  
  │       schema_version: 1.2.0  
  │       provider: OUTBOX  
  │       duration: 3ms  
  │       │  
  │       └── [span] conduit.handle: order.create (order-service)  
  │               attempt: 1  
  │               duration: 47ms  
  │               │  
  │               └── [span] conduit.dispatch: payment.charge  
  │                       causation_id: op-456  
  │                       provider: KAFKA  
  │                       └── [span] conduit.handle: payment.charge  
```  

`trace_id`, `causation_id` и `correlation_id` пробрасываются через `metadata` конверта автоматически.

## 12. Модель безопасности  

### 12.1 Выбор механизма аутентификации  

Conduit поддерживает три механизма. Они **не применяются одновременно** — выбирается один в зависимости от топологии:  

```  
┌──────────────────────────────────────────────────────────────┐  
│  Топология              │  Рекомендуемый механизм            │  
├─────────────────────────┼────────────────────────────────────┤  
│  In-process (InMemory)  │  Нет — доверенная среда           │  
├─────────────────────────┼────────────────────────────────────┤  
│  Межсервисный, своя     │  mTLS на сетевом уровне.           │  
│  инфраструктура         │  Сертификаты управляются через     │  
│                         │  service mesh (Istio / Linkerd)    │  
├─────────────────────────┼────────────────────────────────────┤  
│  Межсервисный, публичный│  JWT в headers конверта.           │  
│  или managed-среда      │  Верифицируется роутером           │  
│                         │  получателя                        │  
├─────────────────────────┼────────────────────────────────────┤  
│  Высокочувствительные   │  mTLS + HMAC подпись payload.      │  
│  данные (PII, финансы)  │  Двойная защита: канал + контент   │  
└─────────────────────────┴────────────────────────────────────┘  
```  

При конфликте (mTLS успешен, JWT истёк) — операция отклоняется. Более строгая проверка имеет приоритет.  

### 12.2 Авторизация  

```  
┌─────────────────────────────────────────────────────────┐  
│  ALLOW  api-gateway    →  order.*        (COMMAND)      │  
│  ALLOW  order-service  →  payment.charge (COMMAND)      │  
│  ALLOW  *              →  *.created      (EVENT, read)  │  
│  DENY   *              →  admin.*        (ALL)           │  
└─────────────────────────────────────────────────────────┘  
```  

Правила применяются в порядке объявления. Первое совпавшее — решающее. `DENY` без условий в конце — рекомендуемый дефолт.  

### 12.3 Шифрование payload  

Для операций с PII или финансовыми данными — шифрование payload на уровне конверта, прозрачное для роутера:  

```  
  Отправитель:   encrypt(payload, recipient_public_key)  
  Роутер:        маршрутизирует зашифрованный конверт  
  Обработчик:    decrypt(payload, own_private_key)  
```  

Роутер не имеет доступа к расшифрованному payload. Это изолирует инфраструктурный слой от данных.

## 13. Эволюция архитектуры  

### 13.1 Монолит → Сервисы без изменения кода  

```  
  Шаг 1. Монолит. Провайдер: InMemory.  
  ┌───────────────────────────────────────┐  
  │  Single Process                       │  
  │  ┌──────────┐  Conduit  ┌──────────┐ │  
  │  │  Order   │──────────►│ Payment  │ │  
  │  │  Module  │ InMemory  │  Module  │ │  
  │  └──────────┘           └──────────┘ │  
  └───────────────────────────────────────┘  
  dispatch(PaymentCharge{...})  

  Шаг 2. Payment выносится в сервис. Провайдер: Kafka.  
  ┌──────────────────┐         ┌─────────────────────┐  
  │  Order Service   │ Conduit │  Payment Service     │  
  │  ┌────────────┐  │────────►│  ┌───────────────┐  │  
  │  │   Order    │  │  Kafka  │  │    Payment    │  │  
  │  │   Module   │  │         │  │    Module     │  │  
  │  └────────────┘  │         │  └───────────────┘  │  
  └──────────────────┘         └─────────────────────┘  
  dispatch(PaymentCharge{...})  ← тот же самый вызов  
```  

Изменилась только конфигурация маршрута: `InMemory` → `Kafka`.  

### 13.2 Рекомендуемый путь эволюции  

```  
┌─────────────────────────────────────────────────────────────┐  
│  Шаг 1. Старт                                               │  
│  In-process. InMemory провайдер.                            │  
│  Нет внешних зависимостей. Быстрые тесты.                   │  
├─────────────────────────────────────────────────────────────┤  
│  Шаг 2. Надёжность                                          │  
│  Критичные команды и события → Outbox/DB.                   │  
│  Новая зависимость: только таблица в существующей БД.       │  
├─────────────────────────────────────────────────────────────┤  
│  Шаг 3. Масштабирование                                     │  
│  Высоконагруженные маршруты → Kafka / NATS JetStream.       │  
│  Брокер подключается точечно, не глобально.                 │  
├─────────────────────────────────────────────────────────────┤  
│  Шаг 4. Декомпозиция                                        │  
│  Модули выносятся в сервисы.                                │  
│  Меняется конфигурация маршрутов.                           │  
│  Бизнес-код не изменяется.                                  │  
└─────────────────────────────────────────────────────────────┘  
```

## 15. Область неприменимости  

| Сценарий | Причина | Альтернатива |  
|---|---|---|  
| Запросы данных (Query/RPC) | Conduit — только write path и уведомления | gRPC, REST, прямой in-process вызов |  
| Streaming (видео, телеметрия) | Conduit — про операции, не потоки байт | Kafka Streams, WebSocket |  
| CRUD API для внешних клиентов | Conduit — внутренний фреймворк | REST / GraphQL / gRPC |  
| Real-time UI обновления | Push-модель вне области Conduit | WebSocket, SSE |  
| Batch-обработка больших объёмов | Не оптимизирован под bulk | Spark, Flink, batch jobs |

## 16. Итог  

**Conduit — операционная шина backend-архитектуры.**  

Он стандартизирует доставку команд и событий, абстрагирует физическую топологию и встраивает надёжность как часть контракта — с честной семантикой, предсказуемым поведением и без ложных обещаний.  

```  
  Без Conduit:                       С Conduit:  

  Монолит ──rewrite──► Сервисы       Монолит ──config──► Сервисы  
  (переписывание коммуникаций)       (конфигурация маршрутов)  

  Race condition в Outbox            Outbox — часть контракта  

  "Exactly-once" как маркетинг       At-least-once + явная  
                                     идемпотентность обработчика  
```
